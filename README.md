**MCP Server** — ядро оркестратора для запуска и координации ИИ-агентов, построенное на чистом **Python 3.10+** без сторонних фреймворков. Каждый агент работает как независимый подпроцесс, общающийся с ядром по простому JSON-протоколу через stdin/stdout. Ядро хранит общую память агентов, обращается к локальной LLM через **Ollama REST API**, поддерживает живой интерактивный диалог и умеет автоматически прогонять запрос пользователя по полной цепочке **Аналитик → Разработчик → Тестировщик**, включая автоматическое исправление кода при падении тестов.

## 🚀 Ключевые особенности

### 🏗 Архитектура
- **Subprocess-агенты** — каждый агент — это отдельный процесс (любой язык, не только Python), ядро не знает о его внутренней реализации.
- **Единый JSON-протокол** — все сообщения между ядром и агентом — это одна строка JSON на stdin/stdout, без привязки к конкретному фреймворку.
- **Общая память агентов** — простое key-value хранилище в памяти ядра с автосохранением на диск, через которое аналитик передаёт ТЗ разработчику, а разработчик — список файлов тестировщику.

### ⚡ Неблокирующее чтение и устойчивость к Windows-кодировкам
- Чтение **stdout** и **stderr** каждого агента вынесено в отдельные потоки (`threading.Thread` + `queue.Queue`) — ядро не "замораживается", ожидая ответ медленного агента, а диагностика агента видна сразу в консоли с префиксом `[agent:<имя>][stderr]`.
- Подпроцессы агентов запускаются с принудительным `PYTHONIOENCODING=utf-8` и `encoding="utf-8"` на чтении — без этого на Windows (где консоль часто использует устаревшую кодовую страницу типа cp1251) агент падал с `UnicodeEncodeError`, если модель генерировала любой нестандартный юникод-символ (например `│`).

### 🤖 Интеграция с LLM
- Прямой REST-клиент к **Ollama** (`/api/generate`, `/api/tags`) без сторонних SDK — как на уровне ядра (`Orchestrator.query_ollama`), так и внутри агентов.
- Никакой заглушки (FakeLLM) — если Ollama недоступна, не запущена или не успела ответить, агент возвращает настоящее, понятное сообщение об ошибке.
- Принудительный JSON-режим Ollama (`"format": "json"`) для запросов ТЗ — заставляет модель генерировать синтаксически валидный JSON на уровне декодирования, а не только "по просьбе" в тексте промпта.

### 💬 Живой диалог с автоматическим переключением режима
- `run_analyst.py` запускает агента один раз и дальше позволяет переписываться с ним в реальном времени.
- Детектор намерения **на уровне кода** (`wants_spec()` в `run_analyst.py`): если в сообщении есть слова "тз" / "техническое задание" / "план проекта" — оркестратор сам включает строгий режим валидации, не полагаясь только на то, что модель поймёт это из системного промпта.

### 🔗 Полный цикл Аналитик → Разработчик → Тестировщик
- При успешной валидации ТЗ оркестратор **автоматически** запускает агента-разработчика, передаёт ему ТЗ и записывает сгенерированный код на диск.
- Сразу после этого **автоматически** запускается агент-тестировщик: генерирует юнит-тесты для каждого файла и прогоняет их через `pytest`.
- Если тесты падают — оркестратор сам отправляет разработчику отчёт об ошибке с просьбой исправить код **в той же папке проекта**, и снова прогоняет тесты. Цикл повторяется до успеха или до исчерпания лимита попыток (по умолчанию 2) — без единого ручного шага.

### 📝 Наблюдаемость
- Логирование всех ключевых событий ядра одновременно в консоль и в файл с разбивкой по дате.
- Отдельный лог невалидных JSON-ответов LLM — `logs/invalid_llm_responses.log`.
- Отдельный лог файловых операций — `logs/fs_operations.log` (что записано, что заблокировано, что упало с ошибкой).
- Отдельный лог прогонов pytest — `logs/test_runs.log` (сколько тестов прошло/упало, код возврата).
- При таймауте ожидания разработчика ядро само заглядывает в папку проекта и показывает, что уже успело записаться на диск — процесс агента не убивается при таймауте, он может продолжать работать в фоне.

## 🧩 Реализованные компоненты

### ✅ Компонент 1 — `Orchestrator` (`core/orchestrator.py`)
Ядро системы, отвечающее за весь жизненный цикл агентов, памяти и цепочки Аналитик→Разработчик→Тестировщик.

- Запуск/остановка агентов как подпроцессов (`start_agent`, `stop_agent`, `stop_all_agents`), с принудительной UTF-8 кодировкой.
- Общая память с персистентностью в `workspace/memory.json` (`remember`, `recall`, `append_history`).
- Приём пользовательского ввода и маршрутизация к первому агенту в цепочке (`handle_user_input`, опционально с `parse_as_spec=True`).
- `_run_pipeline(spec)` — верхнеуровневый цикл: сборка → тесты → (при падении) исправление → тесты снова, с лимитом попыток.
- `_ensure_agent_running()` — общий helper для запуска/переиспользования любого агента по имени файла.
- Таймауты ожидания ответа разработчика и тестировщика масштабируются по числу файлов.
- Диагностика частично готового проекта при таймауте (`_report_partial_project`).
- REST-клиент к Ollama: генерация текста и список установленных моделей (`query_ollama`, `list_ollama_models`).

### ✅ Компонент 2 — `AgentHandle` (`core/orchestrator.py`)
Внутренняя обёртка над каждым запущенным агентом.

- Фоновые потоки для неблокирующего чтения **stdout** (ответы по протоколу) и **stderr** (диагностика, печатается в консоль ядра).
- Очередь сообщений (`queue.Queue`) между потоком-читателем stdout и основным циклом ядра.
- Корректное завершение процесса с фолбэком на `kill()`, если `terminate()` не успел за таймаут.

### ✅ Компонент 3 — `test_agent` (`agents/test_agent.py`)
Минимальный агент-заглушка для проверки протокола — эхо-ответ с порядковым номером сообщения.

### ✅ Компонент 4 — `analyst` (`agents/analyst.py`)
По умолчанию — обычный разговорный ассистент; по запросу ("составь ТЗ") — генератор структурированного ТЗ.

- Читает системный промпт из `workspace/prompts/analyst.md`, хост и модель — из `workspace/configs/config.json`.
- При флаге `want_spec` от ядра передаёт в Ollama `"format": "json"` — принудительная JSON-генерация.
- **Без заглушки**: при ошибке связи с Ollama возвращает настоящее сообщение с причиной (`ollama_unreachable`, `ollama_timeout`, `ollama_error`).
- Работает как постоянный процесс — обрабатывает сообщения одно за другим без перезапуска.

### ✅ Компонент 5 — `parser.py` (`core/parser.py`)
Валидатор структурированных JSON-ответов LLM.

- Извлекает JSON-объект из текста ответа (даже из markdown-блока ```json``` или с пояснениями вокруг).
- **Нормализует названия полей**: если модель перевела ключи на русский (`"Название"`, `"Описание"`, `"Требования"` и т.п. вместо `title`/`description`/`requirements`) — приводит их к ожидаемой схеме автоматически, прежде чем проверять обязательные поля.
- Проверяет обязательные поля (`title`, `description`, `requirements`, `modules`) и типы списков (`requirements`, `modules`, `tech_stack`, `files`).
- Проверяет, что каждый элемент `files` — объект с полем `path` (нужен разработчику).
- Логирует каждый невалидный ответ в `logs/invalid_llm_responses.log`.

### ✅ Компонент 6 — `developer` (`agents/developer.py`)
Агент-разработчик: превращает ТЗ в реальные файлы кода, умеет дорабатывать их по отчёту тестировщика.

- Получает ТЗ либо явно в сообщении (`{"type": "build_request", "spec": {...}}`), либо читает его сам из `workspace/memory.json` как резерв.
- Итерируется по списку `spec["files"]`, для каждого файла формирует контекстный промпт (название и описание проекта, требования, стек, путь и назначение конкретного файла) и обращается к Ollama.
- **Поддерживает `fix_request`**: при сообщении `{"type": "fix_request", "project_dir": "...", "test_report": {...}}` перегенерирует файлы **в той же папке проекта**, добавляя в промпт каждого файла вывод pytest и явную просьбу исправить причину падения, не удаляя нужную функциональность.
- `clean_code_response()` убирает markdown-обёртку (```код```) и типичные вступительные фразы ("Вот код:", "Конечно, держи:"), оставляя только сам код.
- Транслитерирует название проекта в безопасное имя папки (кириллица → латиница) и защищает от перезатирания одноимённых проектов при первой сборке (добавляет таймстамп при коллизии).
- Пишет файлы через `core/fs_tools.py` — безопасно, в `workspace/projects/<slug>/`.

### ✅ Компонент 7 — `fs_tools.py` (`core/fs_tools.py`)
MCP-инструменты файловой системы для разработчика и тестировщика.

- `write_file(relative_path, content)` и `create_directory(relative_path)` — обе операции работают строго в пределах `workspace/projects/`.
- Защита от выхода за пределы (`PathEscapeError`): блокирует `../../etc/passwd`, абсолютные пути и любые попытки резолва пути за границы `workspace/projects/`.
- Каждая операция (успех/блокировка/ошибка) логируется в `logs/fs_operations.log`.

### ✅ Компонент 8 — `tester` (`agents/tester.py`)
Агент-тестировщик: генерирует юнит-тесты для кода разработчика и прогоняет их.

- Получает `project_dir` и список сгенерированных файлов (`{"type": "test_request", "project_dir": "...", "files_written": [...]}`), либо берёт последний проект из общей памяти как резерв.
- Для каждого `.py`-файла читает исходный код и просит Ollama сгенерировать тесты на основе `workspace/prompts/tester.md`.
- Перед тестовым кодом модели детерминированно добавляет заголовок: `import pytest`, путь к родительской папке, **и** `import <module>`, **и** `from <module> import *` — чтобы работали оба стиля вызова (`module.func()` и `func()` напрямую), а также `pytest.raises`/фикстуры.
- Пишет тесты в `<project_dir>/tests/test_<module>.py` через `core/fs_tools.py`.
- Запускает `core/mcp_tools.run_pytest()` и возвращает оркестратору количество пройденных/упавших тестов и хвост вывода pytest.

### ✅ Компонент 9 — `mcp_tools.py` (`core/mcp_tools.py`)
MCP-инструменты для запуска тестов.

- `find_project_python()` — ищет интерпретатор в виртуальном окружении самого сгенерированного проекта (`venv/`, `.venv/`), если разработчик его создал; иначе использует окружение самого MCP-сервера (там уже есть `pytest` из `requirements.txt`).
- `run_pytest()` — запускает `python -m pytest` с таймаутом, разбирает итоговую строку вывода (`"N passed, M failed in X.XXs"`) в количества `passed`/`failed`/`collection_errors`.
- Каждый прогон логируется в `logs/test_runs.log`.

## 🏗 Архитектура

### Поток данных: пользователь → аналитик → разработчик → тестировщик → (исправление)

```
┌──────────────┐ handle_user_input() ┌────────────────┐  stdin (JSON)  ┌──────────┐
│ Пользователь │─────────────────────►│  Orchestrator  │───────────────►│ analyst  │
└──────────────┘                     │                │                │(subprocess)│
                                      │  + memory.json │◄───────────────┤          │
                                      └───────┬────────┘  stdout (JSON) └──────────┘
                                              │  ТЗ валидно (parse_as_spec=True)
                                              ▼
                                      ┌────────────────┐ build_request  ┌──────────┐
                                      │ _run_pipeline() │───────────────►│developer │
                                      │                │◄───────────────┤          │
                                      └───────┬────────┘  files_written └────┬─────┘
                                              │                              │ write_file()
                                              ▼                              ▼
                                      ┌────────────────┐ test_request   workspace/projects/<slug>/
                                      │                │───────────────►┌──────────┐
                                      │ (тот же цикл)   │◄───────────────┤  tester  │
                                      └───────┬────────┘  passed/failed └────┬─────┘
                                              │                              │ run_pytest()
                                   failed > 0 │                              ▼
                                              ▼                        pytest-отчёт
                                      fix_request → developer (повтор, до 2 попыток)
```

Зависимости идут в одну сторону: **пользователь → ядро → агент**. Все три агента (аналитик, разработчик, тестировщик) сами обращаются к Ollama напрямую; ядро дополнительно может обращаться к Ollama самостоятельно через `query_ollama()`.

### Почему именно подпроцессы, а не потоки/корутины

| Критерий | Subprocess-агенты | Потоки в одном процессе |
|---|---|---|
| **Изоляция сбоев** | ✅ Падение агента не убивает ядро | ❌ Необработанное исключение может повредить процесс |
| **Язык реализации** | ✅ Агент может быть на любом языке | ❌ Только Python |
| **Простота протокола** | ✅ JSON-строка — универсальный интерфейс | ⚠️ Нужны общие объекты Python |
| **Параллелизм** | ✅ Реальный параллелизм ОС | ⚠️ GIL ограничивает чистый Python |

## 🔌 Протокол общения ядра с агентом

Ядро → агент (`stdin`):
```json
{"type": "user_input", "text": "Составь ТЗ для бота напоминаний", "memory_snapshot": [...], "want_spec": true}
```

Агент-аналитик → ядро (`stdout`), готовое ТЗ:
```json
{"type": "agent_response", "agent": "analyst", "text": "{\"title\": \"...\", \"files\": [...]}", "source": "ollama"}
```

Ядро → агент-разработчик (`stdin`), первичная сборка:
```json
{"type": "build_request", "spec": {"title": "...", "files": [{"path": "main.py", "description": "..."}]}}
```

Ядро → агент-разработчик (`stdin`), запрос на исправление:
```json
{"type": "fix_request", "spec": {...}, "project_dir": "bot_napominaniy", "test_report": {"passed": 1, "failed": 2, "raw_output": "..."}}
```

Агент-разработчик → ядро (`stdout`):
```json
{"type": "agent_response", "agent": "developer", "project_dir": "bot_napominaniy", "files_written": ["bot_napominaniy/main.py"], "errors": []}
```

Ядро → агент-тестировщик (`stdin`):
```json
{"type": "test_request", "project_dir": "bot_napominaniy", "files_written": [...], "spec": {...}}
```

Агент-тестировщик → ядро (`stdout`):
```json
{"type": "agent_response", "agent": "tester", "project_dir": "bot_napominaniy", "tests_written": ["bot_napominaniy/tests/test_main.py"], "passed": 3, "failed": 1, "raw_output": "..."}
```

Завершение работы агента:
```json
{"type": "shutdown"}
```

Полный список правил — в [`workspace/rules/rules.md`](workspace/rules/rules.md).

## 🛠 Требования

- Python 3.10+

```bash
git clone https://github.com/yushmeh/MCP-Server
cd MCP-Server
pip install -r requirements.txt
```

```bash
ollama serve

>>> /bye
```

## 💬 Интерактивный чат с агентом

```bash
python run_analyst.py
```

Запускает агента один раз и открывает живой диалог. Для выхода — `exit`, `quit` или `выход` (либо `Ctrl+C`).

```
Агент готов к диалогу. Для выхода набери: exit / quit / выход

Вы: привет
Агент: Привет! Как я могу тебе помочь?

Вы: выход
```

Если Ollama недоступна — агент вернёт настоящее сообщение об ошибке, и диалог можно продолжать дальше.

## 🔗 Цепочка Аналитик → Разработчик → Тестировщик

Если в сообщении есть слова **"тз"**, **"техническое задание"** или **"план проекта"** — `run_analyst.py` сам (на уровне кода, не надеясь на модель) включает строгий режим:

```
Вы: Составь ТЗ для телеграм-бота напоминаний о приёме лекарств

🛠 Запускаю агента-разработчика для «...»...
✅ Сгенерировано 5 файлов в workspace/projects/<slug>/
🧪 Запускаю агента-тестировщика для «<slug>»...
   Результат тестов: 4 пройдено, 1 упало
🔁 Тесты упали (1). Передаю отчёт об ошибке разработчику с запросом «исправь ошибку» (попытка 1/2)...
✅ Сгенерировано 5 файлов в workspace/projects/<slug>/
🧪 Запускаю агента-тестировщика для «<slug>»...
   Результат тестов: 5 пройдено, 0 упало
✅ Все тесты прошли успешно для «<slug>». Задача завершена.
```

Что происходит за кадром (метод `Orchestrator._run_pipeline`):
1. Оркестратор просит Ollama сгенерировать ТЗ с `"format": "json"` (принудительный JSON), `core/parser.py` валидирует и нормализует ответ;
2. При успехе ТЗ сохраняется в память под ключом `last_tech_spec`, и запускается `developer.py` — генерирует код по `spec["files"]`;
3. Сразу запускается `tester.py` — генерирует тесты для каждого файла и прогоняет их через `core/mcp_tools.run_pytest()`;
4. Если все тесты прошли — пайплайн завершается с сообщением об успехе;
5. Если есть провалы — оркестратор отправляет разработчику `fix_request` с выводом pytest, тот перегенерирует файлы **в той же папке**, и тестировщик прогоняет тесты заново;
6. Цикл повторяется до успеха или до `max_fix_attempts` (по умолчанию 2) — после чего пайплайн честно останавливается с сообщением о необходимости ручной проверки, без зацикливания;
7. Таймауты ожидания разработчика и тестировщика масштабируются по числу файлов — генерация на CPU без GPU может занимать по 1-2 минуты на файл;
8. Если таймаут всё же наступил на шаге сборки — оркестратор заглядывает в папку проекта и показывает, что уже успело записаться (процесс агента не убивается, может дописывать файлы в фоне).

Чтобы вызвать цепочку программно, минуя чат:
```python
response = orch.handle_user_input("Составь ТЗ для ...", parse_as_spec=True)
```

## 🗂 Схема ТЗ и безопасность файловой системы

Полная JSON-схема, которую генерирует аналитик и потребляют разработчик и тестировщик:

```json
{
  "title": "Короткое название проекта",
  "description": "Развёрнутое описание задачи",
  "requirements": ["Требование 1", "Требование 2"],
  "modules": ["Модуль 1", "Модуль 2"],
  "tech_stack": ["Технология 1", "Технология 2"],
  "files": [
    {"path": "main.py", "description": "Точка входа приложения"},
    {"path": "models.py", "description": "Модели данных"}
  ]
}
```

`title`, `description`, `requirements`, `modules` — обязательны; `files` обязателен для запуска разработчика (без него используется резервный единственный `main.py`).

Все файлы, которые пишут разработчик и тестировщик, проходят через `core/fs_tools.py`:
- путь резолвится и проверяется, что остаётся строго внутри `workspace/projects/`;
- абсолютные пути и `../../` за пределы песочницы — блокируются (`PathEscapeError`);
- каждая операция логируется в `logs/fs_operations.log`.

Тесты тестировщика дополнительно прогоняются через `core/mcp_tools.run_pytest()` — на лучшем доступном интерпретаторе (venv проекта или окружение MCP-сервера), с логированием в `logs/test_runs.log`.

## 📚 Методы Orchestrator

| Метод | Назначение |
|---|---|
| `start_agent(name, script_path)` | Запустить агента как подпроцесс (UTF-8 кодировка) |
| `stop_agent(name)` / `stop_all_agents()` | Остановить одного агента или всех |
| `handle_user_input(text, parse_as_spec=False)` | Передать ввод первому агенту; при `True` — валидировать как ТЗ и запустить полный пайплайн |
| `remember(key, value)` / `recall(key)` | Запись/чтение общей памяти агентов |
| `append_history(role, content)` | Добавить запись в общую историю диалога |
| `query_ollama(prompt, model=...)` | Запрос к локальной LLM через Ollama (на уровне ядра) |
| `list_ollama_models()` | Список установленных в Ollama моделей |
| `_handle_analyst_response(response)` | Валидация ТЗ через `core.parser`, сохранение в память, запуск `_run_pipeline` |
| `_run_pipeline(spec, max_fix_attempts=2)` | Полный цикл: сборка → тесты → исправление → тесты, до успеха или лимита попыток |
| `_ensure_agent_running(name, script)` | Запустить/переиспользовать любого агента по имени файла |
| `_trigger_developer(spec)` / `_trigger_developer_fix(...)` | Первичная сборка / повторная сборка с отчётом об ошибке |
| `_trigger_tester(project_dir, files, spec)` | Запуск тестировщика для уже собранного проекта |
| `_print_dev_result(response)` | Печать результата сборки (файлы/ошибки) |
| `_report_partial_project(spec)` | При таймауте — показать, что уже записано на диск |
| `shutdown()` | Корректно остановить всех агентов и сохранить память |
