# 🧠 MCP SERVER — Ядро оркестратора ИИ-агентов

[![Python](https://img.shields.io/badge/Python-3.10+-3776AB?style=for-the-badge&logo=python&logoColor=white)](https://python.org)
[![Ollama](https://img.shields.io/badge/LLM-Ollama-000000?style=for-the-badge&logo=ollama&logoColor=white)](https://ollama.com)
[![Architecture](https://img.shields.io/badge/Architecture-Orchestrator%20%2F%20Subprocess-00D4FF?style=for-the-badge)](#-архитектура)
[![Protocol](https://img.shields.io/badge/Protocol-JSON%20over%20stdio-39FF14?style=for-the-badge)](#-протокол-общения-ядра-с-агентом)

**MCP Server** — ядро оркестратора для запуска и координации ИИ-агентов, построенное на чистом **Python 3.10+** без сторонних фреймворков. Каждый агент работает как независимый подпроцесс, общающийся с ядром по простому JSON-протоколу через stdin/stdout. Ядро хранит общую память агентов, умеет обращаться к локальной LLM через **Ollama REST API** и поддерживает живой интерактивный диалог с агентом.

## 🚀 Ключевые особенности

### 🏗 Архитектура
- **Subprocess-агенты** — каждый агент — это отдельный процесс (любой язык, не только Python), ядро не знает о его внутренней реализации.
- **Единый JSON-протокол** — все сообщения между ядром и агентом — это одна строка JSON на stdin/stdout, без привязки к конкретному фреймворку.
- **Общая память агентов** — простое key-value хранилище в памяти ядра с автосохранением на диск.

### ⚡ Неблокирующее чтение
- Чтение **stdout** каждого агента вынесено в отдельный поток (`threading.Thread` + `queue.Queue`), поэтому ядро не "замораживается", ожидая ответ медленного агента.
- Чтение **stderr** агента тоже идёт в отдельном потоке и сразу печатается в консоль ядра с префиксом `[agent:<имя>][stderr]` — диагностика ошибок агента (например, проблемы с Ollama) видна сразу, а не теряется в трубе.

### 🤖 Интеграция с LLM
- Прямой REST-клиент к **Ollama** (`/api/generate`, `/api/tags`) без сторонних SDK — как на уровне ядра (`Orchestrator.query_ollama`), так и внутри агента (`agents/analyst.py`).
- Никакой заглушки (FakeLLM) — если Ollama недоступна, не запущена или не успела ответить, агент возвращает настоящее, понятное сообщение об ошибке.

### 💬 Живой диалог
- `run_analyst.py` запускает агента один раз и дальше позволяет переписываться с ним в реальном времени, сообщение за сообщением, без перезапуска процесса.

### 📝 Наблюдаемость
- Логирование всех ключевых событий ядра (запуск/остановка агентов, ввод пользователя, ошибки) одновременно в консоль и в файл с разбивкой по дате.
- Отдельный лог невалидных JSON-ответов LLM — `logs/invalid_llm_responses.log` (используется в опциональном режиме ТЗ).

## 🧩 Реализованные компоненты

### ✅ Компонент 1 — `Orchestrator` (`core/orchestrator.py`)
Ядро системы, отвечающее за весь жизненный цикл агентов и памяти.

- Запуск/остановка агентов как подпроцессов (`start_agent`, `stop_agent`, `stop_all_agents`).
- Общая память с персистентностью в `workspace/memory.json` (`remember`, `recall`, `append_history`).
- Приём пользовательского ввода и маршрутизация к первому агенту в цепочке (`handle_user_input`).
- REST-клиент к Ollama: генерация текста и список установленных моделей (`query_ollama`, `list_ollama_models`).
- Логирование в файл и консоль с автоматическим созданием директорий.

### ✅ Компонент 2 — `AgentHandle` (`core/orchestrator.py`)
Внутренняя обёртка над каждым запущенным агентом.

- Фоновый поток для неблокирующего чтения **stdout** (ответы агента по протоколу).
- Фоновый поток для чтения **stderr** агента — печатается в консоль ядра с префиксом `[agent:<имя>][stderr]`, чтобы ошибки/диагностика агента не терялись.
- Очередь сообщений (`queue.Queue`) между потоком-читателем stdout и основным циклом ядра.
- Корректное завершение процесса с фолбэком на `kill()`, если `terminate()` не успел за таймаут.

### ✅ Компонент 3 — `test_agent` (`agents/test_agent.py`)
Минимальный агент-заглушка для проверки протокола.

- Построчно читает JSON из stdin.
- Отвечает эхо-сообщением с порядковым номером и количеством полученных элементов истории.
- Корректно завершается по сообщению `{"type": "shutdown"}`.

### ✅ Компонент 4 — ИИ-агент (`agents/analyst.py`)
По умолчанию — обычный разговорный ассистент, отвечающий через Ollama.

- Читает системный промпт из `workspace/prompts/analyst.md`.
- Берёт хост и модель Ollama из `workspace/configs/config.json`.
- Обращается к Ollama через REST API (`/api/generate`) с системным промптом и сообщением пользователя.
- **Без заглушки**: при ошибке связи с Ollama агент возвращает настоящее сообщение об ошибке с указанием причины (`ollama_unreachable`, `ollama_timeout`, `ollama_error`) — никакого подделанного ответа.
- Работает как постоянный процесс — обрабатывает сообщения одно за другим без перезапуска, поэтому подходит для живого диалога.
- По умолчанию отвечает свободным текстом; структурированное ТЗ выдаёт только если пользователь явно попросил (см. [режим ТЗ](#-режим-структурированного-тз-опционально)).

### ✅ Компонент 5 — `parser.py` (`core/parser.py`)
Валидатор структурированных JSON-ответов LLM — используется только в опциональном режиме ТЗ.

- Извлекает JSON-объект из текста ответа модели (даже если он обёрнут в markdown-блок ```json``` или сопровождён пояснениями).
- Проверяет обязательные поля схемы ТЗ: `title`, `description`, `requirements`, `modules`.
- Проверяет, что `requirements`, `modules`, `tech_stack` — списки, а не строки/числа.
- Логирует каждый невалидный ответ в `logs/invalid_llm_responses.log` с таймстампом, списком ошибок и исходным текстом модели.

## 🏗 Архитектура

### Поток данных между пользователем, ядром и агентом

```
┌──────────────┐   handle_user_input()   ┌────────────────┐   stdin (JSON)   ┌────────────┐
│ Пользователь │ ───────────────────────►│  Orchestrator  │ ────────────────►│   Агент    │
└──────────────┘                         │                │                  │ (subprocess)│
                                          │  + memory.json │ ◄────────────────┤            │
                                          └────────────────┘   stdout (JSON)  └────────────┘
                                                  │                                  │
                                                  │ REST API                         │ stderr (диагностика)
                                                  ▼                                  ▼
                                          ┌────────────────┐                  консоль ядра
                                          │     Ollama     │
                                          │ localhost:11434│
                                          └────────────────┘
```

Зависимости идут в одну сторону: **пользователь → ядро → агент**. Агент сам обращается к Ollama напрямую (для диалогового режима); ядро дополнительно умеет обращаться к Ollama самостоятельно через `query_ollama()` — например, для других сценариев, не завязанных на конкретного агента.

### Почему именно подпроцессы, а не потоки/корутины

| Критерий | Subprocess-агенты | Потоки в одном процессе |
|---|---|---|
| **Изоляция сбоев** | ✅ Падение агента не убивает ядро | ❌ Необработанное исключение может повредить процесс |
| **Язык реализации** | ✅ Агент может быть на любом языке | ❌ Только Python |
| **Простота протокола** | ✅ JSON-строка — универсальный интерфейс | ⚠️ Нужны общие объекты Python |
| **Параллелизм** | ✅ Реальный параллелизм ОС | ⚠️ GIL ограничивает чистый Python |

## 🔌 Протокол общения ядра с агентом

Ядро → агент (`stdin`):
```json
{"type": "user_input", "text": "Привет!", "memory_snapshot": [/* последние записи истории */]}
```

Агент → ядро (`stdout`), обычный диалог:
```json
{"type": "agent_response", "agent": "analyst", "text": "Привет! Как я могу помочь?", "source": "ollama"}
```

Агент → ядро (`stdout`), ошибка связи с Ollama:
```json
{"type": "agent_response", "agent": "analyst", "error": "ollama_unreachable", "text": "Не удалось подключиться к Ollama..."}
```

Завершение работы агента:
```json
{"type": "shutdown"}
```

Полный список правил — в [`workspace/rules/rules.md`](workspace/rules/rules.md).

## ⚙️ Установка

```bash
git clone https://github.com/yushmeh/MCP-Server
cd MCP-Server
pip install -r requirements.txt
```

Подготовка Ollama:
```bash
ollama serve
ollama pull llama3
```

Перед первым реальным диалогом рекомендуется "прогреть" модель — первый запрос к свежезагруженной модели может занимать заметно больше времени:
```bash
ollama run llama3
>>> Привет
# дождись полного ответа, затем
>>> /bye
```

## 💬 Интерактивный чат с агентом

```bash
python run_analyst.py
```

Запускает агента один раз и открывает живой диалог: можно писать сообщение за сообщением, без перезапуска скрипта. Для выхода — `exit`, `quit` или `выход` (либо `Ctrl+C`).

Пример сессии:
```
Агент готов к диалогу. Для выхода набери: exit / quit / выход

Вы: привет
Агент: Привет! Как я могу тебе помочь?

Вы: как тебя зовут?
Агент: Меня зовут... (ответ зависит от модели)

Вы: выход
```

Если Ollama недоступна, не запущена или не успела ответить — агент вернёт настоящее сообщение об ошибке, и диалог можно продолжать дальше: достаточно поднять Ollama и написать следующее сообщение.

## 📐 Режим структурированного ТЗ (опционально)

Изначально агент задумывался как **аналитик**, превращающий запрос пользователя в техническое задание в формате JSON. Эта возможность не удалена, а сделана **опциональной** — по умолчанию агент ведёт обычный диалог, а строгий JSON-формат включается отдельно.

Чтобы включить режим ТЗ программно, вызови `handle_user_input` с флагом `parse_as_spec=True`:

```python
response = orch.handle_user_input("Хочу телеграм-бота для напоминаний", parse_as_spec=True)
```

В этом режиме:
1. Ядро дополнительно прогоняет ответ агента через `core/parser.py`;
2. Если ответ — валидный JSON с полями `title`, `description`, `requirements`, `modules` — он сохраняется в общую память под ключом `last_tech_spec`, а в консоль печатается `✅ Аналитик подготовил ТЗ «...»`;
3. Если ответ невалиден — печатается `❌ Ответ не прошёл валидацию: [...]`, а сырой ответ логируется в `logs/invalid_llm_responses.log`.

Системный промпт (`workspace/prompts/analyst.md`) уже учитывает оба режима: для обычных сообщений модель отвечает свободным текстом, а при явной просьбе составить ТЗ — переходит на структурированный JSON-формат.

## 📚 Методы Orchestrator

| Метод | Назначение |
|---|---|
| `start_agent(name, script_path)` | Запустить агента как подпроцесс |
| `stop_agent(name)` / `stop_all_agents()` | Остановить одного агента или всех |
| `handle_user_input(text, parse_as_spec=False)` | Передать ввод пользователя первому агенту; при `parse_as_spec=True` — дополнительно валидировать ответ как ТЗ |
| `remember(key, value)` / `recall(key)` | Запись/чтение общей памяти агентов |
| `append_history(role, content)` | Добавить запись в общую историю диалога |
| `query_ollama(prompt, model=...)` | Запрос к локальной LLM через Ollama (на уровне ядра) |
| `list_ollama_models()` | Список установленных в Ollama моделей |
| `_handle_analyst_response(response)` | Валидация ответа через `core.parser` и сохранение ТЗ в память (вызывается только при `parse_as_spec=True`) |
| `shutdown()` | Корректно остановить всех агентов и сохранить память |
